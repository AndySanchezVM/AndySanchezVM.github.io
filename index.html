Etapa 1: Propuesta del proyecto de servicio web

Breve descripción de la organización (giro/sector que atiende, posición en el mercado)

Yo trabajo para un Microsoft Partner quien ayuda a organizaciones en su camino de adoptación de las nuevas tecnologías y hacia la cuarta transformación. Somos proveedores de servicios profesionales para soluciones de Microsoft y todo tipo de licencias de uso de su software. No tenemos un sector al cual nos enfocamos y por lo mismo tenemos clientes dentro del gobierno, escuelas, non-profits, leyes, espacio y contadores.

Impacto de los servicios web en la organización

Los servicios web están en cada departamento de nuestra organización debido a que la compañía ha estado basada en home office desde su inicio en el 2010. Como Microsoft Partners, utilizamos los servicios en la nube de Microsoft 365 y Azure. Esto significa que nuestras tareas diarias ahora están disponibles para colaboración desde nuestros hogares y los documentos están guardados en SharePoint Online y OneDrive for Business donde podemos trabajar con usando las aplicaciones de Office o Microsoft Teams. Nuestro servicio de telefonía igual está integrado a Teams y este sirve como nuestro soft client. Otro ejemplo del uso de servicios web es nuestra herramienta de Customer Relationship Management, usamos un servicio llamado Zoho. Creo que desde la pandemia que empezó el 2020 por COVID-19 el trabajo ha aumentado considerablemente porque muchas compañías quieren mudarse a la nube y permitir que sus empleados trabajen remotamente.

Descripción de la situación o problema a intervenir a través del establecimiento de un servicio web

Existe una brecha entre nuestros departamentos de ventas e ingeniería que me gustaría llenar. La problemática es que después de cerrar una venta, la misma se convierte en proyecto al cual se le da seguimiento a través de una herramienta completamente diferente y existe muy poca visualización a esta información. En mi proposición de solución a este problema, incluiré una herramienta basada en servicios web pero que si tiene integración con el resto de nuestro sistema (Zoho). Con esta integración se podrá dar más visualización para que no se pierda de vista el proyecto de ambos lados y darle el seguimiento correspondiente.

Descripción de la metodología o pasos generales en el desarrollo de un servicio web

Aquí están un ejemplo de los pasos estándar para la creación de los servicios web y una descripción de cada uno.


  • Paso 1. Cree un servicio web

    Escribir el código o utilizar una aplicación para crear el servicio web.

    Paso 2. Cree un WSDL y XSD del servicio web

    Crear el archivo .bat para el WSDL y XSD del servicio web donde se encuentran todos los archivos a ejecutar.

    Paso 3. Publique el servicio web

    Publicar el servicio web para que esté disponible a través de un URL.

    Paso 4. Cree un cliente de servicio web

    Escribir el código o utilizar una aplicación para crear el cliente que invocara al servicio web.

    Paso 5. Cree el servicio web con soporte WS-Security

    Para ofrecer un servicio web seguro, se establecerá una combinación de usuario y contraseña. Esto permite proporcionar autenticación, confidencialidad, e integridad de los servicios web.

    Paso 6. Cree la clase que expone el servicio de seguridad

    Crear el archivo donde se identifican cada opción o perfil de configuración de seguridad disponible.

    Paso 7. Cree el descriptor del servicio de seguridad a desplegar

    Crear el controlador del servicio de seguridad que validara si el usuario llamando al servicio tiene el acceso permitido.

    Paso 8. Cree la clase de comprobación de usuario y contraseña

    Crear el controlador que se encargue de resolver esa cuestión de seguridad de validar la contraseña contra el nombre de usuario.

    Paso 9. Despliegue el servicio web con soporte WS-Security

    Empaquetar el servicio web con el soporte de WS-Security.

    Paso 10. Cree el cliente para el servicio web de seguridad

    Crear el cliente que se encargara de pedir, validar y dar acceso al servicio web.

    Paso 11. Ejecute las pruebas unitarias al servicio web de seguridad

    Para garantizar el lanzamiento exitoso del servicio web y sus componentes de seguridad es importante realizar pruebas. Así se pueden identificar problemas y corregirlos antes del lanzamiento oficial.

    Paso 12. Realice el empaquetado y ejecución del servicio cliente

    Para finalizar, se empaqueta todo el programa de servicio web y su respectivo cliente que lo acezara.

    Justificación de la arquitectura a utilizar de acuerdo con los requerimientos de la organización

    Según Altvater la diferencia más grande entre SOAP y REST es que “REST accede a los datos mientras que SOAP realiza operaciones a través de un conjunto más estandarizado de patrones de mensajería. Para mi solución a la problemática identificada creo que seria de mas beneficio la arquitectura de REST por algunos de los beneficios mencionados por el autor. Entre estos esta que REST acepta una mayor variedad de formatos de datos mientras que SOAP solo permite XML y el punto mas importante es que “REST es más rápido y utiliza menos ancho de banda, lo cual es esencial para trabajo remoto. También es más fácil de integrar con sitios web existentes sin necesidad de refactorizar la infraestructura del sitio.”



    Etapa 2: Diseño de arquitectura del servicio web

    Definición de fuentes de datos a utilizar

    Las fuentes de datos necesarias para este proyecto son nuestra herramienta de Customer Relationship Management (CRM) llamada ZohoCRM en donde se puede encontrar toda la información de nuestros clientes existentes y potenciales. Aquí también se ubica la información sobre los proyectos que se han contratados por los clientes.

    Procesos de extracción, transformación y carga de datos

    En este caso, y debido a que los datos con los cuales se estará trabajando ya están viviendo en la base de datos de ZohoCRM no se ocupa ningún proceso de extracción, transformación o carga de datos. El único requerimiento será darle seguimiento a la Deal que se crea por el departamento de ventas o probablemente crear alguna nueva manera de continuar con esta Deal postventa.

    Mecanismos e infraestructura para el almacenamiento y gestión de datos

    Como mencione, todo esto sería un web hosting de parte de nuestro proveedor de SaaS, Zoho, quien creo ZohoCRM. Por esta razón, no se ocupa ninguna infraestructura adicional ni que preocuparnos sobre el almacenamiento de datos. Seria mas que nada un cambio de procesos y políticas dentro de la empresa y más específicamente, dentro de estos dos departamentos (ventas e ingeniería).

    Interfaz de usuario a utilizar

    La interfaz de usuario es extremadamente amigable y muy fácil de aprender. Los de ventas ya están entrenados a usar CRM solo sería cuestión de entrenar a los ingenieros para que puedan seguir con su parte después de que los de venta cierren las Deals.

    Mecanismos de seguridad

    Zoho es una herramienta en la nube muy robusta y debido a que es un SaaS, su acceso está basada en licencias por usuario. Hago mención de esto porque es importante el usuario solamente pude acceder al servicio (o herramienta) según la licencia que se le a asignado. Otros mecanismos de seguridad incluyen, el control de acceso por permisos y activar el multi-factor authentication o MFA.
